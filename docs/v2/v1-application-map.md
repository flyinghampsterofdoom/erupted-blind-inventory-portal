# V1 application map

## Scope and method

This inventory describes the server-rendered V1 application as implemented on 2026-07-15. It was derived from `app/main.py`, the three V1 routers, all V1 Jinja templates, all 72 SQLAlchemy models, `sql/schema.sql`, services, scripts, configuration, and tests. V2 routes and assets are excluded except where V1/V2 coexistence matters. “Admin” below means the effective check used by the route, which is not always the same as the literal `ADMIN` role; see the permission map.

V1 is a FastAPI/Jinja/PostgreSQL application. Most mutations are HTML form POSTs protected by a double-submit CSRF cookie. Cash reconciliation and barcode receiving use JSON-like fetch endpoints. There is no task queue, scheduler, webhook, SPA, or separate public API.

## Cross-cutting observations

- Store/location identity is the central join between local workflows and Square. Most store workflows enforce scope by using `principal.store_id`; management filters generally accept any active store.
- The application has two inventory-count implementations: rotating blind sessions and admin full-store counts. Both snapshot quantities and can push Square, but use separate tables/services.
- “Current state plus immutable history” is repeated for change box, master safe, cash reconciliation, chores, and ordering. V2 must preserve both sides of each pair.
- Many defaults are created lazily during GET/service execution (dashboard categories, checklist items, chore tasks, inventory/par rows, ordering settings). Read-only-looking routes may therefore write.
- There is no general deletion policy. Some catalogs use `active`; several drafts/history rows are hard-deleted by explicit routes.
- Tests concentrate on ordering math/receiving and inventory reports. Most authentication, permissions, cash, forms, chores, audits, and route behavior are untested.
