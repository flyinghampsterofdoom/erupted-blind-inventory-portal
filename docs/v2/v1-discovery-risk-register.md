# V1 discovery risk register

Ratings describe V2 parity/migration risk, not a claim of an active incident.

| Rating | Risk | Evidence | Potential impact | Required mitigation/validation |
|---|---|---|---|---|
| Critical | Square inventory writes are spread across four services with different idempotency behavior | Count/admin-count use fresh UUIDs; emergency includes UUID; receiving uses deterministic keys | Duplicate or partial on-hand changes during retries/cutover | Inventory-write gateway, frozen integration tests, reconciliation runbook, explicit cutover lock |
| Critical | Store par delivery mutates two inventory domains in one workflow | Changes change-box current quantities and creates submitted non-sellable take from a staged queue | Cash/non-sellable divergence or lost delivery facts | Transaction-level parity tests and dual-domain reconciliation before migration |
| High | Startup and bootstrap mutate schema without migration history | Startup DDL; idempotent 1,400-line schema with ALTERs/triggers; no version table | Environment drift, unsafe rollback, unknown deployed schema | Capture production schema diff/read-only; introduce versioned baseline before V2 schema work |
| High | Literal ADMIN checks bypass capability overrides | Multiple reports and Square push routes use `require_role` | Denied users may retain sensitive access; MANAGER parity mismatch | Effective-access test matrix; decision gate before changing behavior |
| High | Historical reports use current Square/mapping/cost data | COGS, vendor sales, valuation, analytics recalculate live | V2 and V1 differ for same historical date; non-reproducible audits | Record golden responses and decide snapshot policy |
| High | Rotating count/recount is a hidden state machine | Rotation, forced count, snapshots, draft, submit refresh, recount queue, 3-match auto-push | Lost recounts, wrong closeouts, accidental Square writes | State-transition model and exhaustive integration tests before rebuild |
| High | PO receiving can partially succeed externally | Multiple per-line/store Square writes inside request; success/failure event mix | Local PO status and Square inventory disagree | Reconciliation screen/runbook, idempotent command model, failure injection tests |
| High | Local filesystem is authoritative for PO PDFs | DB stores relative path under instance-local `generated/` | Missing files across hosts/deploys; stale PDF served | Inventory existing files; move to managed storage or deterministic regeneration contract |
| High | Demo seed runs on every bootstrap with known example identities | Bootstrap always calls `app.seed_example` | Example users/data can appear in an operational DB | Environment-gate seed; audit existing principals before any rollout |
| High | Tests do not cover most workflows or permissions | 50 V1 tests focus on ordering/reports/recount math; no route/auth/cash/form/chore tests | Silent parity/security regressions | Characterization tests before each module migration |
| Medium | Current state has multiple writers | Change inventory, non-sellable stock, principals, mappings | Last-write-wins behavior differs after modularization | Assign explicit data ownership and command boundaries |
| Medium | Dashboard visibility is not authorization | Role/category/card filters differ from backend checks | Hidden accessible pages or visible denied cards | Test navigation and backend enforcement separately |
| Medium | Management unlock operations are broadly permitted | Count and non-sellable unlock use `management.access` | LEAD can reopen submitted records | Preserve initially; raise explicit authorization decision |
| Medium | Customer request aggregate semantics are ambiguous | Submission lines are facts; `request_count` is separately overwritable | Double counting or loss if derived incorrectly | Compare production usage; define whether count is derived or curated |
| Medium | Cash expected values are live while verification stores partial snapshots | Square corrections/timezone data can change | Stored actual/history appears to disagree with current expected | Preserve per-verification expected and document recomputation rules |
| Medium | Text fields act as enums | `purchase_orders.invoice_payment_status`, sync `sync_type`, delivery `item_type`, form `section`, audit `action`, metadata keys | Invalid/novel values and migration mapping gaps | Inventory distinct production values before constraining |
| Medium | Null and zero encode different meanings inconsistently | PO costs/pars/received qty, cash expected, inventory price/cost, manual vs suggested pars | Calculations and UI states change in V2 | Field-by-field null/zero characterization fixtures |
| Medium | Enum/status models contain apparently unused states/tables | Receipt tables and `RECEIVED_SPLIT_PENDING`, `COMPLETED`, `CANCELLED`; vendor contacts/email fields | Removing “unused” paths may break operational/manual data | Query production references before retirement; mark Investigate |
| Medium | Hard deletes remove operational records | Session purge, admin draft, chore draft, change-box count, PO draft | Audit/history gaps or orphan assumptions | FK/cascade audit and retention decision before exposing V2 deletes |
| Medium | Read-looking services can write defaults | Dashboard, chores, checklists, inventory/par/settings initialization | Read-only V2 pages change data or fail parity if initialization omitted | Separate initialization from queries after characterization |
| Medium | No centralized Square retry/rate-limit handling | Many urllib clients and 3600 s timeout | Hung requests, inconsistent error UX, rate-limit amplification | Shared client/observability design after parity baseline |
| Medium | Store identity lacks ORM-visible uniqueness | SQL schema makes `square_location_id` unique; model declaration does not | Test/in-memory metadata and DB behavior differ | Treat deployed SQL constraint as authoritative; align only in approved schema milestone |
| Medium | Count session and forced count have cyclic FKs | SQLAlchemy warns it cannot sort the two tables | Migration ordering and cascade complexity | Explicit constraint ordering and data audit |
| Medium | Session token stored unhashed | Raw bearer token in `web_sessions` | DB disclosure yields active sessions | Security redesign decision; preserve behavior only as temporary compatibility |
| Medium | CSRF/session cookies depend on environment flags | Example recommends secure true; local defaults false | Misconfiguration weakens transport protection | Deployment config validation; no secrets in repo/docs |
| Low | Autosave liveness timer only initializes on autosave pages | Base JS returns early if no autosave forms | Non-autosave page remains open after session expiry until next action | Characterize UX; unify in V2 shell if approved |
| Low | GET export writes audit log | Session CSV route commits | Caching/read-only assumptions fail | Preserve audit intentionally or move to explicit export command |
| Low | Placeholder and compatibility routes may look unused | Audit Queue placeholder; opening-checklist POST-URL GET redirect | Premature removal breaks bookmarks/mobile refresh | Redirect/telemetry period during cutover |
| Low | Template JavaScript embeds business calculations | Change/safe totals, PO allocation/receiving, cash UI | Server/client rounding or validation mismatch | Golden browser tests plus server-authoritative validation |

## Weakest coverage areas

No repository tests were found for authentication/session/CSRF, capability overrides, dashboard configuration, store scoping, count routes and Square push integration, admin full count, cash reconciliation, chores, opening checklist, change box/form/safe/par delivery, non-sellable stock, customer forms, exchange returns, employee logs, user administration, schema bootstrap, PDF/CSV/XLSX routes, or failure/retry behavior. These areas require characterization before implementation, not just after.
